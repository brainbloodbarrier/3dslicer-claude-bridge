# V2 Roadmap

## Why This Exists

This file is the lightweight home for the v2 effort.

The goal is to improve spine workflows and agentic MCP usage for Claude Code and
Cursor without creating a heavy planning framework. Existing markdown files stay
the primary documentation surface. This roadmap only captures the direction and
the main slices of work.

## Working Rules

- Prefer updating existing docs over creating new docs.
- Keep the active task list in `TODO.md`.
- Keep `CLAUDE.md` as the main place that points to active project guidance.
- If a new markdown file is ever needed, add it for a concrete reason and link
  it from `CLAUDE.md`.

## V2 Direction

### 1. Keep the current core, improve the workflow layer

Do not rewrite the entire bridge from scratch.

Reuse the current Slicer transport, validation, and proven imaging logic, while
improving the workflow layer around them.

### 2. Make spine workflows the main focus

The first v2 target is the current spine surface:

- pre-op spine evaluation
- fracture evaluation
- Modic and MRI workflows
- CCJ workflows
- cervical screw planning
- construct comparison

### 3. Reduce drift between docs and the real MCP surface

The current `.claude/commands/*` workflows are useful, but some no longer match
the true tool signatures and prerequisites. V2 should tighten that gap instead
of adding more layers around it.

### 4. Improve Claude Code and Cursor parity

High-value workflows should be usable through MCP itself, not only through
Claude-local command files. Cursor should be able to benefit from the same
workflow surface.

## Near-Term Work

### Phase 1: Align and simplify

- audit existing spine workflow docs against real MCP tool signatures
- identify the highest-value workflow mismatches
- clean up the current docs so they describe reality

### Phase 2: Define the v2 workflow surface

- decide which spine workflows deserve first-class MCP workflow tools
- separate low-level primitives from higher-level orchestration
- improve discoverability for both Claude Code and Cursor

### Phase 3: Migrate carefully

- migrate one workflow at a time
- test each migrated workflow against current behavior
- avoid broad rewrites without proving the new slice works

## Non-Goals For Now

- building a large ADR/spec/issues system
- rewriting the entire codebase before testing new workflow slices
- introducing many new docs before the need is proven

## Success Looks Like

- the repo has one clear place to understand the v2 direction
- the active work is visible in `TODO.md`
- spine workflows become easier to use and less drift-prone
- Claude Code and Cursor share more of the same MCP-native workflow surface
